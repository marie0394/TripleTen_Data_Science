SEED = 33333
def upsample(data, target, repeat):
    data_zeros = data[data[target] == 0]
    data_ones = data[data[target] == 1]
    
    repeat_int = int(repeat)
    repeat_dec = repeat - repeat_int
    
 

    data_upsampled = pd.concat([data_zeros] +
                               [data_ones] * repeat_int +
                               [data_ones[0:int(repeat_dec*len(data_ones))]])


    data_upsampled = shuffle(data_upsampled, random_state=SEED)

    return data_upsampled

def find_metrics(y_true, y_pred, title = 'Métricas'):
    accuracy = accuracy_score(y_true, y_pred)
    roc_auc = roc_auc_score(y_true, y_pred)
    df = pd.DataFrame(data={'Precision': [accuracy], 
                           'ROC-AUC': [roc_auc]})
    df.columns = pd.MultiIndex.from_product([[title], df.columns])
    # df.style.set_caption(title)
    display(df)
    return [accuracy, roc_auc]

Conclusiones¶

Los modelos 2, 3 y 4 mostraron que están sobreajustados. Hay una diferencia de aproximadamente 0.06-0.09 entre las métricas de la dataset de prueba y entrenamiento. Aun así, sus predicciones superan al de la prueba de cordura, por lo que servirían como modelos de predicción. El modelo 1 fue el único que no tuvo sobreajuste según las métricas resultantes, sin embargo, también tiene el menor valor de AUC-ROC, por lo que no sería el modelo idea a elegir.

Nos interesa el valor de la métrica, pero también nos interesa que el modelo sea confiable para predecir. Podríamos decir que el modelo 4, CatBoost, sería el elegido, ya que fue el que resultó con el mejor ROC-AUC. Sin embargo, también tiene la mayor diferencia entre los resultados del dataset de prueba vs el entrenamiento. Indicaría que es el modelo con mayor problema de sobreajuste de los probados.

Dado esto, la elección final sería con el modelo 2, que usó bosque aleatorio, ya que dió un ROC-AUC de 0.896, el segundo más alto, teniendo también de los menores sobreajustes, lo que nos da más confianza de que el modelo sirva para predecir datos nuevos.

Conclusiones¶

Se comenzó con 4 archivos con información de los contratos y los clientes, los cuales fueron unidos a través del CustomerID. Se realizaron cambios a los tipos de variables de un par de columnas antes de continuar con la unión. Se creó la variables ContractEnded la cual sería nuestra variablle objetivo.

Luego se realizó el preprocesamiento, primero se rellenaron los datos ausentes. Como estos valores ausentes provienieron de las tablas de los planes de internet y teléfono, se asumió que esto implicaba que no estába usando dicho servicio un usuario en específico. Por tanto, se rellenó con 'No' estas variables para indicar que el usuario no tiene esos servicios activos. Luego se calculó una variable extra que contendría el número de meses que ha tenido un usuario el contrato.

La mayoría de datos eran categóricos binarios, por lo que fueron reemplazados por valores de 1/0. Luego se quitaron las columnas que no serían necesarias para el entrenamiento. Dado que la relación entre la clase positiva y negativa de la función objetivo estaba desbalanceada, se realizó sobremuestreo, para equipar mejor la proporción de ambos.

Varios modelos requieren solo usar datos numéricos por lo que se aplicó One-Hot Encoding para tratar los valores categóricos restantes.

Además de la prueba de cordura, se crearon 4 modelos, uno que aplicó regresión logística, otro de bosque aleatorio, uno con LightGBM y otro de CatBoost. Los 4 superaron la prueba de cordura y los mejores valors de AUC-ROC se obtuvieron del modelo 2 y 4. Aunque el modelo 4 tenía la mejor métrica, también presenta el mayor sobreajuste. Por tanto, se eligió el modelo 2, que tiene aun así el segundo mejor AUC-ROC pero tiene menor sobreajuste, dando más confianza que este modelo pueda predecir mejor en casos de datos nuevos.
